## Day 1 — 12/18 (목) : **"문장을 분해하는 데 반나절을 쓰다"**

### 아침 ~ 오전

이번 과제의 문장은 **"기업의 업무에 도움이 되는 AI 기반 서비스"** 였습니다.

많은 사람들이 이런 문장을 받으면 바로 **"어떤 서비스를 만들지"** 에 뛰어들어요.
저는 다르게 시작했습니다. **"이 문장이 실제로 의미하는 게 뭘까"** 를 먼저 분해했습니다.

> 아이디어를 먼저 정하고 문제를 맞추는 방식이 아니라,
> 문제를 해석하는 과정 속에서 아이디어가 뒤따라오는 흐름.

### 오후 — 기획 밀도에 베팅하다

이전 프로젝트 경험을 통해 **"기획부터 배포까지 한 서비스를 완주하는 데 필요한 최소 시간"** 을 알고 있었어요. 그걸 기준으로 역산해보니, 이번 4일에서는 **구현 속도보다 기획 밀도를 높이는 편이 더 효과적**이었습니다.

의도적으로 목요일 오후의 대부분을 **기획에만** 썼습니다.

### 질문 릴레이

- "업무에 도움이 된다는 건 어떤 형태로 나뉘나?"
- "효율 개선 말고, 사고 정리/아이디어 생산/자본 운영/스트레스 관리 같은 영역은?"
- "AI가 가장 자연스럽게 개입할 수 있는 지점은 어디?"
- "반복적인 작업인가, 사고 과정을 단축할 수 있는 영역인가?"
- "실제 업무에서 불필요하게 시간과 사고를 소모하는 지점은?"

이 질문들을 한 줄씩 써내려가다가, **결론에 도달했습니다.**

> 문제는 **"업무 성과를 높이는 것"** 이 아니다.
> 문제는 **"업무를 정리하고 인식하는 과정에서 발생하는 사고 비용"** 이다.

### 저녁 — LAYOUTNEMO의 컨셉이 잡히다

문제가 좁아지니, 해결의 방향이 자연스럽게 따라왔습니다.

기존 도구(노션, 지라)가 **단방향 트리 구조** 로 업무를 담는 반면, 실제 사고는 **다차원 그래프** 에 가깝다는 인식. 그래서 **"사고를 그대로 펼쳐놓을 수 있는 캔버스 기반 개인 사고 공간"** 이 필요하다는 결론에 도달했어요.

**저녁까지 정해진 것들:**

- 서비스 컨셉: 비주얼 캔버스 기반 사고 공간
- AI의 역할: "대신 판단하지 않는다. 흐름이 끊기지 않게 돕는다"
- AI 온/오프 구조를 **처음부터** 포함
- "로컬 우선" 을 기본 원칙으로

**Day 1 성과**: 아직 코드 한 줄 없음. 그러나 **"무엇을 만들지"** 가 분명해진 하루.

## Day 2 — 12/19 (금) : **"v0가 하루 만에 뱉어낸 것"**

### 오전 — Vercel v0 로 뼈대 뽑기

하루 종일 기획만 할 순 없었습니다. 구현은 빠르게 들어가야 했고, 그 시작은 **Vercel v0** 였습니다.

v0는 AI로 UI를 자동 생성해주는 도구입니다. 기획 단계에서 정리한 개념들 — 블럭, 영역, 관계선, 시급도, 캔버스 — 을 자연어로 입력했더니 **반나절 만에 기본 UI 뼈대가 튀어나왔어요.**

### v0 의 장점

- 초기 UI/기능 구조를 **시각적으로 빠르게** 확인할 수 있음
- 핵심 사용 시나리오를 **조기 검증** 가능
- "이게 말이 되는 서비스인가" 를 코드 짜기 전에 느껴볼 수 있음

### v0 의 한계 (곧바로 체감됨)

v0는 **"그럴듯한 첫 삽"** 을 떠주지만, **세부 로직의 일관성** 은 보장하지 않았습니다.

- 블럭 이동은 되지만 자동 연결이 이상하게 작동
- "정리하기" 버튼은 있는데 실제 동작은 단순 mock
- 시급도 색상이 일관성 없이 혼재
- 드래그 이벤트가 완료 영역과 충돌

이걸 그대로 서비스로 낼 수는 없었어요.

### 오후 — GitHub 이관 후 VSCode 에서 손으로 정제 시작

v0 결과물을 GitHub 레포로 옮기고 VSCode 에서 직접 코드를 다루기 시작했습니다.
이 순간부터 **"개발자의 개입"** 이 본격적으로 필요해졌어요.

#### 이날 밤까지 정리한 것들

- 블럭 드래그 + 자동 저장 구조
- 기본 시급도 색상을 통일
- 캔버스 상태를 localStorage 에 직렬화하는 로직
- Undo 기본 스켈레톤 (이후 `skipHistory` 플래그로 정제됨 — 별도 Devlog 예정)

**Day 2 성과**: 뼈대가 섰지만, 아직 **"서비스 같지는 않은"** 상태.

## Day 3 — 12/20 (토) : **"AI와 싸운 날"**

토요일은 **AI 프롬프트에 가장 많은 시간을 쓴 날** 이었습니다.

### 오전 — AI 연동의 첫 벽: Vercel AI Gateway

v0가 붙여둔 AI 호출은 Vercel AI Gateway 를 경유하는 구조였습니다. 편리해 보였지만, 실제로 **결제 카드 등록을 요구** 했어요. 데모 단계에서 이걸 진행하기엔 불필요한 장벽이었습니다.

**판단: Vercel AI Gateway → OpenAI API 직접 호출로 전환.**

API 키를 `.env.local` 에 두고, 서버 사이드 라우트(`/api/ai/create-block`, `/api/ai/tidy-comprehensive`) 에서 직접 OpenAI 를 호출하는 구조로 바꿨어요. 모델은 `gpt-4o-mini` 로 결정. 빠르고 저렴하면서 JSON 응답 모드까지 지원했습니다.

### 오후 — 프롬프트가 창작을 시작했다

AI가 블럭 생성 시 사용자 입력을 **과하게 창작하는 문제** 가 발견됐습니다. 사용자가 "결제 플로우 다시 생각" 이라고 쓰면, AI는 마음대로 **"결제 플로우 UX 전면 리뉴얼 프로젝트"** 같은 제목으로 부풀렸어요.

이건 이 서비스의 철학과 정면으로 충돌하는 동작이었습니다.
**AI는 판단을 대신하지 않는다** 는 원칙과요.

#### 결국 정착한 5단계 프롬프트 구조

1. **내용 분석** — 사용자 입력 그대로 해석
2. **제목 생성** — 입력 범위를 벗어나지 않는 선에서만
3. **기한 확인** — "내일", "다음 주" 같은 자연어 날짜 파싱
4. **긴급도 확인** — 키워드 기반 urgency 추정
5. **영역 선택** — 기존 영역 중에서만 선택

거기에 **"사용자 입력을 그대로 유지하라"** 는 금지 조항을 명시하고, `temperature: 0.3` 으로 일관성을 확보했습니다.

이 5단계 프레임워크에 대한 더 자세한 얘기는 **별도 Devlog 에서 깊게** 다룰 예정이에요.

### 저녁 — 도메인 구매

서비스 주소로 **`layoutnemo.com`** 을 Namecheap 에서 구매했습니다.

이건 단순 배포 준비를 넘어서 **"추후 확장과 반복 개선을 전제로 한 기준점"** 을 만드는 작업이었어요. 도메인을 직접 소유하면, 인프라를 Vercel → AWS 로 옮기더라도 서비스 주소는 유지할 수 있으니까요.

(이 결정 자체가 하나의 Devlog 글감입니다 — **"도메인을 산 이유 — 기준점을 만든다는 것"** 으로 언젠가 쓰고 싶어요.)

**Day 3 성과**: AI 가 서비스 철학에 맞게 움직이기 시작. 도메인 확보.

## Day 4 — 12/21 (일) : **"배포의 날, 그리고 포기의 날"**

### 오전 — 줌/팬 기능 실험 → 롤백

일요일 아침에 **"캔버스니까 Figma 처럼 줌/팬이 되면 좋지 않을까"** 라는 생각이 들어서 바로 구현을 시도했습니다.

- 스페이스바 누르면 캔버스 이동
- `Cmd + Plus/Minus` 로 확대/축소
- 100% 리셋 버튼

**결과: 전체 기능이 망가졌어요.**

좌표 변환 로직이 들어오자,
- 블럭 드래그가 엉뚱한 위치로 가고
- 완료 영역 감지가 작동하지 않고
- 관계선 클릭 삭제가 좌표 오차로 실패하고
- 자동 연결도 거리 계산이 틀어졌습니다.

4~5 시간을 씨름한 끝에 **"이건 이번 4일 안에 안정화 불가"** 라는 결론에 도달했고, 결국 **전부 롤백** 했습니다.

> 교훈: **좌표 변환 복잡도는 다른 기능 전체에 전염된다.**

이 경험도 **별도 Devlog** 로 깊게 쓸 예정입니다. 이번 4일 전체에서 가장 아픈 시간이었거든요.

### 오후 — 마지막 기능 안정화

줌/팬 실험을 접고, 남은 시간에 **이미 있는 기능들을 안정화** 하는 데 집중했습니다.

- **휴지통 시스템**: "잠시 치워두기" → "삭제" 로 명칭 변경, 10개 제한
- **연결선 클릭 삭제** 의 race condition 수정 (`onBatchUpdateBlocks` 도입)
- **블럭 자동 연결**을 "중심점 거리" 에서 **"실제 겹침 감지"** 로 전환
- **정리하기 다이얼로그** 의 accept/reject 후 무한 fetch 루프 제거
- **키보드 단축키** (Cmd+N, Cmd+Z, Cmd+K, Cmd+F, Cmd+E, Alt+click)
- **JSON / Markdown 내보내기**

이 중 race condition 이슈는 **"연결선 삭제 시 개별 블럭 두 개를 각각 업데이트하다가 충돌"** 이 문제였어요. `onBatchUpdateBlocks` 로 두 블럭을 한 번에 업데이트하도록 바꿔서 해결했습니다.

### 저녁 — Vercel 배포

GitHub `master` 브랜치에 push 하니, Vercel 이 자동으로 빌드하고 `layoutnemo.com` 으로 붙였습니다.

HTTPS 는 자동, www/non-www 둘 다 접속 가능. **일요일 밤 9시쯤 "접속 가능한 상태"** 가 됐어요.

### 밤 — 회고를 쓰다

배포가 끝나고 나서 곧바로 **개발 후기를 쓰기 시작했어요.** 기억이 생생할 때 남겨야 한다는 생각이었습니다. 그 기록이 지금 여러분이 읽고 있는 [1편](./2025-12-21-how-layoutnemo-started) 의 원문이 됐습니다.

**Day 4 성과**: 서비스가 실제 인터넷에 떠 있음. `layoutnemo.com` 접속 시 기능 대부분 정상.

## 전체 회고 — 4일의 의미

### 이 일정이 가능했던 이유

**기획 밀도에 시간을 투자한 덕분** 이라고 생각합니다.
목요일 오후에 충분히 고민했기 때문에, 이후 3일 동안 **"무엇을 만들지"** 에 대한 망설임이 없었어요. 구현 중 갈림길이 나올 때마다 Day 1에 정해둔 원칙 ("AI는 대신 판단하지 않는다", "로컬 우선", "사고를 펼쳐놓는 공간") 으로 되돌아가서 결정할 수 있었습니다.

### 가장 잘한 선택

- **목요일 오후를 기획에만 쓴 것**
- **Vercel v0 로 빠르게 뼈대를 뽑고, 그 뒤에 직접 정제한 것**
- **도메인을 산 것** — 4일짜리 과제에 도메인까지 사는 건 과해 보일 수 있지만, **"기준점을 만든다"** 는 의미에서 결정적이었어요.
- **줌/팬을 롤백한 것** — 고집을 부렸다면 배포 자체가 불가능했을 겁니다.

### 가장 아픈 선택

- **시급도 5 → 4** 로 줄인 순간 명시적 결정이 없었던 것 (자연스럽게 바뀐 건 맞지만, 기록이 안 됐음)
- **`sizeLevel` 완전 포기** — 설계 철학의 일부였는데 시간에 밀려 사라졌음 (다음 버전에서 재설계 필요)
- **데이터 구조 `aiMeta` 계층이 localStorage 에 반영되지 않음** — 행동은 철학대로지만 구조는 단순화됐음

### 지금 돌아보면

**4일은 짧은 시간이지만, 동시에 긴 시간이었습니다.**

짧다고 느끼는 건 "더 많이 만들 수 있었을 텐데" 의 관점이고,
길다고 느끼는 건 **"한 사람이 기획-구현-배포를 혼자서 돌릴 수 있다는 걸 증명한 시간"** 이라는 관점이에요.

이 프로젝트는 완결된 결과물이 아닙니다.
**실험과 개선의 출발점** 이고, 지금 이 Devlog 자체가 그 다음 단계의 시작입니다.
