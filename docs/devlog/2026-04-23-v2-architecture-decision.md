## 시작하며

오늘 하루 코드는 한 줄도 안 쳤다.
그런데도 이 프로젝트에서 **가장 중요한 날** 중 하나였던 것 같다.

데모 버전(12/21) 이후 처음으로 **"이 서비스가 앞으로 어떻게 생겨야 하는가"** 를 한 번 제대로 앉아서 정한 날이었거든.

---

## 오늘의 전제 — 원하는 건 이런 게 아니었다

사실 나는 처음부터 노션처럼 **"로그인 하나로 아무 웹에서나 접속해서 내 작업 이어가는"** 걸 원했다.

근데 현재 LAYOUTNEMO 는:

- localStorage 에만 저장된다.
- 다른 브라우저, 다른 기기로 가면 **내 블럭이 하나도 안 보인다.**
- 구글 로그인은 붙어있는데, 그건 그냥 "로그인만 됐을 뿐" 이지 데이터가 서버에 있지 않다.

4일짜리 데모 버전이라서 충분히 이해는 가는 상황이지만, 이대로 더 쓸 수는 없었다.
**서비스가 서비스답지 않았다.**

---

## 첫 번째 오해 — "EC2 사야 하는 거 아닌가?"

이게 오늘 가장 먼저 깨진 가정이었다.

내 머릿속에선:

> DB 스키마 짜고 → 백엔드 서버 따로 만들고 → AI 서버도 분리하고 → EC2 구매해서 올리고…

이 모든 게 **한 세트로 필요하다**고 생각했어.

근데 Claude 한테 설계 얘기를 꺼내자마자 이 생각이 흔들렸다.

> "Vercel + Supabase 만 있으면 EC2 없이도 전부 가능합니다."

처음엔 믿기지 않았다. 블로그로 쓰는 깃허브 페이지도 아니고, 진짜 데이터 저장하고 로그인 처리하고 AI 호출까지 하는 풀스택 서비스인데?

하나씩 짚어보니 정말 그렇더라:

- **Next.js API Routes** 가 이미 백엔드 역할을 하고 있음.
- **Supabase** 가 Postgres + Auth + Realtime 을 무료로 제공함.
- **OpenAI 호출** 은 Vercel 함수에서 그냥 fetch 로 쏘면 됨. 별도 AI 서버 필요 없음.
- **Row-Level Security** 로 DB 단에서 권한 검증 처리 → 백엔드 로직 최소화.

**EC2 월 몇만 원 아낀 거다.** 그리고 운영 부담(서버 관리, 보안 패치, 스케일링) 도 다 날아갔다.

→ 교훈: **"해봤던 방식" 이 아니라 "지금 있는 도구" 를 다시 점검하자.**

---

## 두 번째 현실 — "AI 를 프리로 열면 내가 망한다"

두 번째로 부딪힌 게 **AI 비용** 이었다.

OpenAI `gpt-4o-mini` 가 싸긴 해도 **유저가 많아지면 내가 감당 못 함**.

노션처럼 일단 무료로 쓰게 해주고 싶은데, 그러려면:

1. 광고 붙이기 (SDK 도입 + 수익 발생 전까진 적자)
2. 프리미엄으로 AI 분리

나는 **2번** 으로 가기로 했다. 이유는:

- 광고는 UX 를 망친다. LAYOUTNEMO 는 "사고를 펼쳐놓는 조용한 공간" 이어야 한다. 광고 띠가 있는 사고 공간은… 상상만 해도 이상하다.
- **AI 는 선택지여야 한다** 는 게 원래 서비스 철학이었다. 데모 때도 AI 토글 OFF 를 처음부터 넣은 이유가 이거였다. 그러면 AI 를 **돈 내고 쓰는 프리미엄** 으로 분리하는 게 철학과도 잘 맞는다.

그래서 결정:

- **프리 유저**: 기본 기능 다 쓸 수 있음. 하지만 **AI 는 비활성**.
- **크레딧**: 종량제 토큰 구매. AI 쓸 때마다 차감.
- **구독**: 월 정액. AI 월 한도 + 캔버스 여러 개 생성 가능.
- **어드민**: 나 (개발/테스트용). 환경 변수로 관리.

결제 시스템(Stripe, 토스페이먼츠) 은 지금 붙이지 않는다. **스키마에 훅만** 남겨두고, 나중에 그 훅에 결제 로직을 꽂는 식. `entitlements` 테이블과 `ai_consumption_log` 테이블을 지금 만들어두면, 나중에 결제 붙일 때 스키마 마이그레이션이 필요 없다.

---

## 세 번째 깨달음 — "결 = 태그였다"

사실 이건 블로그 작업하면서 이미 정리된 내용이긴 한데, 오늘 Supabase 스키마를 그리면서 한 번 더 분명해졌다.

처음엔 **"영역" / "Zone" / "Area"** 가 혼용되고 있었다.
근데 정작 기능을 뜯어보면 **"블럭이 가진 속성 태그, 클릭하면 그 태그를 가진 블럭이 강조됨"** — 즉 **태그 기반 필터 UX** 였다.

"영역" 이라고 부르면 공간적 구획 같은데, 실제 UX 는 공간이 아니라 **메타데이터** 였던 거지.

그래서 Claude 랑 네이밍을 다시 찾았는데, **`결(Facet)`** 이 가장 잘 맞았다.
- **결** — 나무의 결처럼, 하나의 사물이 여러 결을 가진다.
- **Facet** — 다면체의 한 면. 하나의 블럭을 여러 관점에서 볼 수 있다.

이 단어를 쓰자마자 **정체성이 훨씬 선명해졌다.**

그리고 스키마에서는:

- **캔버스 독립** (유저 전역 아님). 각 캔버스가 자기 결을 가진다.
- **기본 5개 시드**: `기획 / 개발 / 디자인 / 마케팅 / 일상`
- Postgres 트리거 `AFTER INSERT ON canvases` 로 자동 생성.

빈 캔버스에서 시작하지 않고, **기본 결이 채워진 상태로 시작** 한다. 이게 신규 유저의 온보딩에도 좋다.

---

## 네 번째 고민 — "놓친 게 있을 것 같은데?"

ERD 그리고 RLS 붙이는 거까진 어찌어찌 했다.
근데 뭔가 찜찜했다. **"이게 끝이 아닌 것 같은데?"**

아키텍처라는 게 단순히 데이터 테이블 몇 개 그리는 게 아니라는 느낌이 왔다.

그래서 Claude 한테 물었다: **"전체 그림에서 내가 개발 지식 부족해서 놓친 거 있으면 찾아줘."**

그러자 20개 정도의 이슈가 쏟아졌다.

- Service Role Key 관리 (NEXT_PUBLIC_ 붙이면 즉사)
- Rate limit 다층화 (분/시/일/월)
- Sentry 에러 추적
- Supabase CLI 로 마이그레이션 관리
- 네트워크 실패 UX (낙관적 업데이트 + 토스트)
- 로그아웃 시 로컬 캐시 클리어
- 첫 유저 튜토리얼 블럭
- 법적 문서 (약관, 개인정보처리방침)
- RLS 보안 회귀 테스트
- 계정 삭제 API
- 데이터 이주권 (import/export)
- …

이 중 몇 개는 **"아, 그래서 서비스들이 그 기능을 기본으로 넣는구나"** 싶은 것들이었다.

예를 들어 **로그아웃 시 로컬 캐시 클리어** — 이거 내가 한 번도 생각해본 적 없는데, 공용 PC에서 다음 사람이 내 블럭 보는 시나리오는 진짜 위험하잖아. 이런 걸 **코딩하다가 발견하는 게 아니라, 설계 단계에서 미리 점검해서 로드맵에 넣는 것** 이 아키텍처 작업이라는 걸 느꼈다.

## 오늘 확정한 것들 정리

### 스택

```
Browser (Next.js) ↔ Vercel (API Routes / Server Actions)
                 ↔ Supabase (Postgres + Auth + Realtime)
                 ↔ OpenAI (AI 엔드포인트)
```

- **EC2 없음**. Vercel + Supabase 로 전부 처리.
- **별도 AI 서버 없음**. OpenAI 호출은 Vercel 함수에서.

### 데이터베이스 6개 테이블

1. `canvases` — 캔버스
2. `zones` — 결 (Facet)
3. `blocks` — 블럭
4. `block_relations` — 관계선 (m:n)
5. `entitlements` — 유저 플랜/권한
6. `ai_consumption_log` — AI 호출 이력

### 과금

- 프리: AI 비활성, 캔버스 1개
- 크레딧: 종량제 토큰
- 구독: 월 정액 + 여러 캔버스
- 어드민: 환경 변수로 내 계정 등록

### 마이그레이션 정책 (게스트 → 로그인)

- 서버에 데이터 없으면 → localStorage 자동 복사
- 양쪽에 데이터 있으면 → 4지 선다 다이얼로그
  - 서버 그대로 쓰기 / 덮어쓰기 / 새 캔버스로 병합 / 버리기

### 배제한 것

- EC2
- 별도 AI 서버
- CRDT 기반 충돌 해결 (LWW 로 충분)
- PWA / 오프라인 모드
- 조직/팀 모델
- Stripe 실연동 (스키마 훅만)

## 오늘의 교훈

### 1. **"해봤던 방식" 의 관성을 의심하자**

EC2 구매 얘기가 제일 대표적이었다. 내가 이전에 배웠던 "서비스 = 서버 구매" 라는 공식이 지금은 오래된 공식일 수 있다. 새 도구(Vercel, Supabase) 를 봤을 땐 **"얼마나 많은 일을 알아서 해주는지"** 를 먼저 확인하자.

### 2. **데이터 스키마가 곧 비즈니스 모델이다**

`entitlements` 테이블을 지금 만들지 안 만들지가 **미래에 결제 붙일 수 있는지** 를 결정한다. 스키마는 단순히 저장 구조가 아니라, **서비스가 할 수 있는 일의 경계** 다.

### 3. **"전체 그림 점검" 을 의도적으로 요청해야 한다**

혼자 짜면 ERD 까지만 나온다. 그런데 그 다음 **"내가 뭘 놓쳤지?"** 라는 질문을 명시적으로 던지지 않으면, 로그아웃 캐시 클리어 같은 이슈들은 코딩 중에 버그로 발견된다. 설계 단계에서 점검하는 게 10배 싸다.

### 4. **코드 한 줄도 없는 하루가 가장 생산적일 수 있다**

오늘 Git 변경 파일은 `ARCHITECTURE.md` 하나. 그런데 이 문서 한 장이 앞으로 몇 주의 코딩을 결정한다.
**설계는 코딩보다 늦어 보이지만, 가장 빠르게 정답에 가까워지는 일이다.**
