## 들어가며

[1편](./2025-12-21-how-layoutnemo-started)에서는 LAYOUTNEMO가 어떤 문제 의식에서 출발했는지, 4일 동안 어떤 판단들을 거쳤는지 이야기했습니다.

이번 글은 조금 다른 각도의 회고입니다.

**"데모 버전에서 기획했던 것과, 실제로 만든 것은 얼마나 달랐는가?"**

결과적으로 이번 편은 **포기한 것들에 대한 기록**에 가깝습니다. 무엇을 포기했는지 자체가 이 서비스의 정체성을 말해주더라고요.

---

## 1. 시급도 — 5단계에서 4단계로

### 🎯 기획안

초기 기획에서는 시급도를 **5단계**로 설정했습니다.

| 단계 | 이름 | 기획한 색상 | 의미 |
|------|------|-------------|------|
| 1 | Urgent (긴급) | 🔴 빨간색 | 즉시 처리 필요 |
| 2 | Normal (보통) | 🔵 파란색 | 일반 업무 |
| 3 | Thinking (고민중) | 🟣 보라색 | 아직 구체화되지 않은 아이디어 |
| 4 | Stable (안정) | ⚪ 회색 | 천천히 진행 가능 |
| 5 | Lingering (방치) | 🟠 주황색 | 미루고 있지만 언젠가 해야 |

이 5단계의 핵심 의도는 **"감정과 상태까지 담는 시급도"** 였습니다.
단순히 "중요도" 가 아니라, 지금 이 일이 머릿속에서 어떤 무게감으로 놓여 있는지를 표현하고 싶었어요.

### 🧱 실제로 남은 것

master 브랜치를 지금 열어보면, 시급도는 **4단계**입니다.

```ts
urgency?: "stable" | "thinking" | "lingering" | "urgent"
```

`normal` 이 빠졌습니다.
그리고 색상 매핑도 기획안과 다르게 정착됐어요.

| 단계 | 실제 구현 색상 | 라벨 |
|------|----------------|------|
| `stable` | ⚪ 회색 | 안정 |
| `thinking` | 🔵 파란색 | 생각 중 |
| `lingering` | 🟡 노란색 | 머물러 있음 |
| `urgent` | 🟠 주황색 | 시급 |

### 왜 이렇게 바뀌었을까

솔직히 고백하자면, **"명시적인 설계 변경"이 아니었어요.**
개발을 진행하면서 **"normal 이 있으면 사용자가 어떤 기준으로 골라야 할지 모호해진다"** 는 감각이 쌓였고, UI를 만들다 보니 4개 쪽이 더 깔끔하게 떨어졌습니다.

- 🔴 빨강 → 🟠 주황 으로 톤 다운한 것은, 다른 색들과 **시각적 조화**를 맞추기 위함이었습니다. 빨강은 너무 소리치는 느낌이었어요.
- 🟣 보라 → 🔵 파랑 (thinking) 으로 바뀐 건, 파랑의 차분한 뉘앙스가 "생각 중" 이라는 상태와 더 맞다고 느꼈기 때문입니다.
- 🟠 주황 → 🟡 노랑 (lingering) 은 "머물러 있음" 의 느낌을 **신호등의 노랑** — 멈춤과 주의 사이의 뉘앙스 — 로 잡고 싶었습니다.

기획의 색상 규칙은 "선명함" 위주였고, 실제 구현은 "함께 놓였을 때의 조화" 위주로 간 셈이에요.

### 남은 숙제

코드를 들여다보면 아직 **라벨 통일이 덜 되어 있습니다.**

- `block-detail-dialog.tsx` → "안정적 / 생각 중 / 머물러 있음 / 시급함"
- `create-block-dialog.tsx` → "안정 / 보통 / 주의 / 시급"

같은 `thinking` 상태를 한 곳에선 "생각 중", 다른 곳에선 "보통" 이라고 부르고 있어요. 4일이라는 시간 안에서 어느 순간 통일 타이밍을 놓친 부분입니다.

블로그에서는 **"안정 / 생각 중 / 머물러 있음 / 시급"** 으로 정식 표기를 통일했고, 코드 쪽도 다음 리팩터링에서 같은 이름으로 맞출 예정입니다.

## 2. 블럭 크기 — 완전히 포기한 개념

### 🎯 기획안 (데이터 구조 설계 문서)

초기 데이터 구조 설계에는 **`sizeLevel`** 이라는 필드가 있었습니다.

```json
{
  "id": "block-101",
  "sizeLevel": 1 | 2 | 3 | 4 | 5,
  "shadowColor": "#374151"
}
```

설계 의도는 명확했어요.

> **크기 = 인지적 무게**
> **shadowColor = 심리적 압박, 신경 쓰임**

이건 단순한 우선순위 표시가 아니라, **시각적 위계 이론**에 가까운 설계였습니다.
"우선순위"가 아니라 **"머릿속에서 차지하는 자리"** 라는 개념을 도입하고 싶었어요.

### 🧱 실제로 구현된 것 — 없음

현재 `WorkBlock` 타입에 **`sizeLevel` 은 존재하지 않습니다.**

```ts
interface WorkBlock {
  // ...
  width: number
  height: number
  // sizeLevel ❌
  // shadowColor ❌
}
```

대신 `width` / `height` 가 직접 저장되고, **모든 블럭은 고정된 크기 (360px)** 로 렌더링됩니다.
세부 내용이 길면 높이만 자동으로 늘어나고, **크기로 중요도를 표현하지 않습니다.**

### 왜 포기했나

**솔직히 말하면, 구현 난이도에 막혔습니다.**

크기로 중요도를 표현하려면:
- 블럭 배치 시 다른 블럭과 안 겹치도록 **크기를 감안한 스마트 배치 로직**이 필요합니다.
- 관계선을 그릴 때 **블럭 경계가 매번 달라지므로** 연결점 계산이 복잡해집니다.
- 자동 연결(가까이 놓으면 선이 이어짐)에서 **"가깝다" 의 기준**이 크기마다 달라집니다.
- 완료 영역 감지도 **크기별로 다르게** 처리해야 합니다.

4일 안에 이걸 다 안정적으로 구현하고, 다른 핵심 기능(정리하기, AI 토글, 자동 연결, 연결선 삭제 등)까지 맞물리게 만들 수 있을지 확신이 서지 않았어요.

**판단은 빠르게 했습니다.**
"크기 대신 색상만으로 중요도를 표현한다. sizeLevel 은 이번 범위에서 포기."

### 포기한 결과, 얻은 것

포기하고 나니 오히려 **메시지가 더 선명해졌습니다.**

- 블럭 크기가 모두 같으니, **"이 일이 저 일보다 더 크다"** 는 폭력적인 시각이 사라졌어요.
- 중요도는 **색상(상태)** 으로만 표현되니까, **"무엇이 중요한지"가 아니라 "지금 어떤 상태인지"** 에 집중하게 됩니다.
- "머물러 있음" 같은 상태를 블럭 크기로 표현하는 건 사실 기획 단계에서부터 좀 어색했어요. 작은 블럭이 머물러 있다는 건 이상하잖아요.

**결과적으로, "크기 기반 우선순위"는 이 서비스의 정체성과 맞지 않았습니다.**
포기해서 후회가 없는 선택이에요.

### 블로그에서 다루지 않는 이유

앞으로 LAYOUTNEMO를 소개하는 공식 문서에서는 **`sizeLevel` 이라는 개념을 아예 언급하지 않습니다.**

이유는 간단해요. **현재 서비스에 존재하지 않는 기능을 설계 문서에 남겨두면, 사용자가 혼란만 받습니다.**

"이런 게 있었는데 사라진 것" 은 이 Devlog 같은 아카이브성 글에만 기록하고, About 문서에서는 **"모든 블럭은 같은 크기"** 라는 현재 사실만 보여주려 합니다.

## 3. 데이터 구조 — 청사진과 현실의 거리

초기 데이터 구조 설계 문서는 이렇게 시작했습니다.

```json
{
  "user": { ... },
  "areas": [ ... ],
  "blocks": [ ... ],
  "uiState": { ... },
  "aiMeta": { ... }
}
```

최상위에 5개 키를 두고, 특히 **`aiMeta`** 라는 AI 전용 공간을 분리한 것이 설계의 핵심이었습니다.

> **AI 판단 결과 ≠ 사용자 확정 값**

이 원칙을 구조로 강제하고 싶었어요. AI가 아무리 똑똑해지더라도, 그 판단이 사용자의 확정값과 섞이지 않게. 그래야 도구가 지배자가 되지 않으니까요.

### 🧱 실제로 구현된 것

실제 localStorage에 저장되는 구조는 훨씬 단순합니다.

```
layout_canvases       // Canvas[] — 모든 캔버스
layout_current_canvas // 현재 선택된 캔버스 ID
```

그리고 `Canvas` 안에는 `blocks[]`, `zones[]` 가 들어있을 뿐, **`aiMeta` 같은 분리된 공간은 없습니다.**

또 필드명도 조금씩 달라졌어요:

| 설계 문서 | 실제 코드 |
|-----------|-----------|
| `areaId` | `zone` |
| `summary` | `description` + `detailedNotes` |
| `sizeLevel` 1~5 | `width` + `height` 직접 |

### 왜 이렇게 됐을까

데모 버전의 가장 큰 제약은 **시간** 이었습니다.

`aiMeta` 를 제대로 분리하려면:
- AI 점수를 저장할 때마다 별도 스키마로 쓰기
- 사용자 확정값과 AI 판단값이 **절대 섞이지 않는 보장 로직**
- AI 판단이 바뀌면 UI가 이를 "제안" 으로만 반영하고, 사용자 수락 시에만 확정값 이동

이걸 다 구현하는 건 4일 범위를 넘어가는 작업이었습니다.
그래서 **"기능으로는 유지하되 (AI 제안 → 사용자가 Accept/Reject), 저장 구조는 단순화"** 하는 방향으로 갔어요.

현재 정리하기 기능은 **메모리 상에서 AI 제안을 들고 있다가, Accept 시에만 블럭 상태를 갱신**하는 방식입니다. `aiMeta` 같은 영구 저장 공간 없이도 **행동 자체는 설계 의도를 유지**하고 있어요.

하지만 **영구 저장된 구조로 이 철학이 강제되는가** 라는 질문에는 "아직 아니다" 라고 답해야 합니다.

### 다음 숙제

이 갭은 **리팩터링으로 좁혀야 할 지점**입니다.

- 블럭 저장 구조에 `aiMeta` 섹션을 분리하고
- 현재 `zone`, `description` 같은 이름을 `areaId`, `summary` 로 통일할지 아니면 코드 이름을 유지할지 결정하고
- 설계 원칙 **"AI 판단과 사용자 확정의 분리"** 를 코드 수준에서 강제하는 구조로 옮겨가는 것

이게 다음 버전에서 할 일이에요.

## 4. 로그인 — 의도적으로 뒤로 미룬 것

데이터 구조 설계에는 `user` 객체가 있었고, 처음부터 **"로그인 붙여도 마이그레이션 불필요한 구조"** 를 의도했습니다.

```json
"user": {
  "id": "local",
  "rolePreset": "PM"
}
```

`id: "local"` 이라는 초기값을 두고, 로그인 시점에 이걸 서버 `userId` 로 교체하는 방식이었어요.

### 실제 구현

현재는 **localStorage 만 쓰고 있고**, 로그인 기능은 이미 **Google OAuth via Supabase** 로 붙어 있습니다 ([Roadmap](../roadmap) 참조).

데모 버전(12/21 시점)에는 로그인이 없었고, **의도적으로 제외**한 선택이었습니다. "이번 단계의 서비스 가치에 얼마나 기여하는가" 라는 기준으로 봤을 때, 로그인 구현에 하루를 쓰느니 **정리하기 기능의 안정화**에 시간을 쓰는 게 맞다고 판단했어요.

### 회고

돌이켜보면 **올바른 결정**이었습니다.

**로그인은 서비스의 본질이 아닙니다.** LAYOUTNEMO의 본질은 "캔버스 위에 블럭을 펼쳐놓는 경험" 이고, 그 경험은 로그인 없이도 100% 제공됩니다. **로컬 우선** 이라는 원칙 자체가 이 서비스의 성격을 말해주기도 하고요.

로그인은 **"여러 기기에서 이어가고 싶을 때 추가로 선택하는 기능"** 이어야지, 필수 진입장벽이 되면 안 됩니다.

## 마치며 — 포기한 것이 정체성을 만든다

4일 안에 많은 것을 포기했습니다.

- 5단계 시급도 → 4단계
- sizeLevel → 전면 폐기
- 설계의 `aiMeta` 계층 → 단순화
- 로그인 → 의도적 보류

**그런데 이상하게도, 포기하고 나니 서비스의 정체성이 더 뚜렷해졌어요.**

- 크기가 아니라 색상으로만 시급도를 표현하니까, 블럭이 **평등**해졌습니다.
- normal 을 빼니까, 시급도 선택이 "감정과 상태" 중심으로 모였습니다.
- 로그인을 미루니까, **"로컬 우선"** 이라는 철학이 자연스럽게 서비스의 출발점이 됐습니다.

**기획이 100% 구현되지 않았다는 것**은 실패가 아니라, 4일이라는 제약 안에서 **"정말 중요한 것이 무엇인가"** 를 걸러낸 과정이었다고 지금은 생각해요.
