## 시작하며

어제는 v2 아키텍처를 설계하는 **"의사결정의 날"** 이었다면, 오늘은 **"손을 걷어붙이고 청소하는 날"** 이었다.

데모 버전(12/21) 이후 손대지 않았던 코드를 처음 제대로 뜯어봤다.
v0 로 뽑고 황급히 손으로 정제했던 코드라, **잔해가 많이 남아있을 거라 짐작은** 했지만
실제로 열어보니 생각보다 많았다.

코드 리뷰 결과 이슈가 **18개**. 그 중 🚨 P0 5개, ⚠️ P1 8개, 💡 P2 5개.

오늘 하루 동안 그 중 **상당수를 치웠고**, 덤으로
**i18n (한영 전환)**, **"완료 → 갈무리" UX 재설계**, **다크 모드 톤 개편** 까지 마쳤다.
코드 한 덩이가 아니라 여러 덩이였어서 결과적으로 꽤 큰 커밋이 되었다.

## 1. 바이브 코딩의 진짜 위험 — `ignoreBuildErrors: true`

`next.config.mjs` 에 이 한 줄이 있었다.

```js
typescript: {
  ignoreBuildErrors: true,
}
```

**TypeScript 에러가 있어도 빌드가 통과한다** 는 뜻이다.
그 말인즉슨, 타입이 실제로 맞지 않아도 **런타임에서야 터진다**.

이 한 줄을 껐더니 **숨어 있던 9개의 타입 에러** 가 줄줄이 나왔다.

- `components/work-block-card.tsx:9` — 존재하지 않는 경로 `@/types/work-block` 을 import 중
- `types/index.ts` 의 `WorkBlock` 에 `isAIControl`, `aiEnabled` 필드 누락. 그런데 코드에서는 버젓이 사용 중
- `lib/ai/types.ts` 의 `CreateBlockAIOutput` 스키마가 **프롬프트의 실제 응답 포맷과 완전히 달랐다.** 타입은 `area / due_date / urgency: "안정" | "보통" | "시급"`, 실제는 `suggestedZone / suggestedDueDate / suggestedUrgency: "stable" | "thinking" | ...`
- `components/header.tsx` 의 `HeaderProps` 에 `onRedo` 필드 누락

특히 `CreateBlockAIOutput` 케이스가 가장 아팠다.
**타입 정의와 런타임 데이터가 서로 완전히 다른 형태로 공존** 하고 있었는데, `ignoreBuildErrors` 하나 때문에 이게 배포까지 갔다.
런타임에 문제가 안 터진 이유는 자바스크립트의 **"객체에 없는 키는 undefined"** 라는 관용 덕분이었을 뿐이다.

> **교훈**: `ignoreBuildErrors` 는 "빠르게 돌리기 위한 트릭" 이 아니라 **"타입 안전망을 통째로 포기하는 선언"** 이다.
> 바이브 코딩으로 빠르게 뽑았다면 그 다음 할 일은 **이 스위치를 끄고 나온 에러를 다 잡는 것**이다.

## 2. 거짓말 고치기

코드가 **스스로 거짓말을 하고 있는 부분** 이 몇 군데 있었다.

### 가이드 블럭의 시급도 설명

사용 설명서 가이드 블럭 안에는 이런 글이 적혀 있었다.

```
• Urgent (빨강): 지금 해야 할 일
• Thinking (주황): 고민이 필요한 일
• Lingering (노랑): 계속 미루고 있는 일
• Stable (초록): 안정적으로 진행 중
```

그런데 실제 구현 색상은,

| 시급도 | 가이드 문구 | 실제 구현 색상 |
|--------|-------------|----------------|
| urgent | 🔴 빨강 | 🟠 **주황** |
| thinking | 🟠 주황 | 🔵 **파랑** |
| lingering | 🟡 노랑 | 🟡 노랑 ✅ |
| stable | 🟢 초록 | ⚪ **회색** |

**4개 중 3개가 거짓말** 이었다.
사용자가 앱을 켜서 처음 읽는 안내 문구가 **실제 화면과 다르다는 것** 은, 서비스 신뢰에 작은 상처다.
고쳤다.

### 시급도 라벨이 컴포넌트마다 달랐다

| 컴포넌트 | `thinking` 의 라벨 |
|----------|--------------------|
| `block-detail-dialog.tsx` | **"생각 중"** |
| `create-block-dialog.tsx` | **"보통"** |

같은 상태를 **두 컴포넌트가 서로 다른 이름으로** 부르고 있었다.
유저 입장에서는 **"보통" 과 "생각 중" 이 같은 건지 다른 건지** 알 수 없다.

해결은 단순했다. `lib/constants/urgency.ts` 라는 **단일 소스 파일** 하나 만들고, 모든 곳에서 그걸 import 하게 한 것.
이런 건 복잡한 리팩터링이 아니라, **처음부터 그렇게 만들었어야 할 것**이다.

## 3. 결(영역) 용어 정리와 i18n 도입

어제 결정해둔 **"영역 → 결(Facet)"** 전환을 실제 UI 문자열에 반영했다.
그리고 그 김에 **한영 언어 토글** 까지 넣었다.

### 언어 토글 버튼: `Aa` 와 `가`

좌상단 로고 옆에 작은 토글 버튼을 뒀다.

- 한국어 모드일 때는 **`Aa`** — 눌러서 영문으로
- 영문 모드일 때는 **`가`** — 눌러서 한국어로

iOS 시스템 키보드의 언어 전환 아이콘에서 따왔다. 짧고, 뭔지 바로 알아본다.

### 번역 범위 = "UI 크롬만"

i18n 의 가장 큰 함정은 **"어디까지 번역할 것인가"** 다.
사용자가 자기 손으로 쓴 블럭 제목을 영문 모드로 바꿨다고 영어로 번역하면 이상하다.

그래서 규칙을 이렇게 잡았다.

- **번역 O** — 버튼, 다이얼로그 타이틀, 라벨, 메뉴, 시급도 이름, 기본 제공 결 이름, 기본 시드 블럭(가이드/예시)
- **번역 X** — 사용자가 작성한 블럭 제목/설명/메모, 사용자가 추가한 결 이름

### "사용자가 편집했는가" 를 어떻게 아는가

기본 시드 블럭(가이드, 예시) 은 언어에 따라 번역돼야 하지만, 사용자가 한 번 편집했다면 그 편집본을 존중해야 한다.
아주 단순한 규칙으로 해결했다.

```ts
function translateSeedBlockField(block, field, language) {
  const seed = SEED_BLOCK_STRINGS[block.id]
  if (!seed) return block[field]
  // 저장된 값이 ko 원본과 정확히 일치하면 "유저가 편집하지 않은 것" 으로 간주 → 번역
  if (block[field] !== seed[field].ko) return block[field]
  return seed[field][language]
}
```

**저장된 값 === 한국어 원본 → 사용자 편집 안 됨 → 번역 가능**
**저장된 값 !== 한국어 원본 → 사용자가 건드린 것 → 원본 유지**

수학적으로 완벽하지는 않다 (사용자가 영문으로 편집하다가 실수로 ko 원본과 같아질 수도 있다).
하지만 **99% 사용 패턴에서 문제 없고**, 스키마에 추가 필드를 붙이지 않아도 되는 **싸고 깔끔한 방식** 이다.

### AI 응답 언어까지 연동

영문 모드에서 AI 가 한국어로 응답하면 어색하다.
그래서 API 엔드포인트에 `language` 필드를 받아서, 프롬프트 끝에 이런 지시문을 동적으로 붙였다.

```
Write all text fields (title, summary, zoneReason) in English. Do not use Korean in the response.
```

작은 한 줄이지만, 영문 유저가 **AI 자연어 입력 → 영문 제목/요약 추출** 을 쓸 수 있게 해준다.

## 4. 다크 모드 톤 개편 — "GitHub Dark" 에서 벗어나기

기존 다크 모드는 순수 **회색 + 검정** 기반이었다. 개발자스럽고 차가웠다.
LAYOUTNEMO의 감성 — "사고를 펼쳐놓는 조용한 공간" — 과는 조금 동떨어진 느낌이었다.

### oklch 의 hue 를 건드렸다

CSS 토큰의 다크 변수를 **중성(hue 0)** 에서 **남색 쪽(hue 265)** 으로 옮겼다.

```css
.dark {
  --background: oklch(0.17 0.012 265);    /* 남색 섞인 딥 블랙 */
  --card: oklch(0.215 0.012 265);          /* 배경보다 밝아서 카드가 떠오름 */
  --foreground: oklch(0.955 0.002 265);    /* 완전 흰색 대신 약간 톤 다운 */
}
```

변화 포인트가 두 가지였다.

1. **전체적으로 살짝 남색 기조** — "Notion Dark" / "Linear Dark" 에 가까운 톤
2. **카드 배경을 배경보다 밝게** — 기존엔 카드 배경과 페이지 배경이 똑같이 어두워서 블럭이 **배경에 녹아드는** 느낌이었다. 카드를 한 단계 밝게 하니 블럭이 **또렷이 떠오른다**.

### 시급도 그림자도 약하게

기존 다크 모드에서는 시급도별 그림자가 **거의 네온 사인** 수준이었다.

```
thinking (파랑) : rgba(..., 0.8)   → 눈부심
lingering (노랑): rgba(..., 0.85)  → 형광펜 같음
urgent (주황)   : rgba(..., 0.9)   → 과포화
```

전부 `0.5 ~ 0.6` 선으로 낮췄다. **후광은 유지하되, 네온은 벗어낸다**.

## 5. 완료 → 갈무리 — 네이밍이 UX 를 바꾼다

가장 많은 고민을 한 부분이다.

### 왜 "완료" 라는 단어가 거슬렸나

LAYOUTNEMO는 **"정리되지 않은 사고를 그대로 펼쳐놓는 공간"** 이 철학이다. 그런데 "완료" 는 **"이제 끝났다"** 라는 종결을 전제하는 단어다. 애매한 상태 — "계속 미루고 있음", "생각 중" — 을 담는 서비스에서 **끝을 강요하는 기능** 이 어색했다.

사용자와 얘기하면서 나온 대안이 **"갈무리"** 였다.

> 갈무리
> - 일거리를 정리하여 마무리함
> - 물건 따위를 잘 정리하거나 간수함

**"마무리" 와 "간수"** 둘 다 뜻에 들어있다.
**"완료 아니고, 잠시 치워둠"** 이라는 뉘앙스가 정확히 우리가 원하는 것이었다.

### 공간 차지 문제 — 우하단 점선 영역 대신 "독 아이콘 + 모달"

기존 UX는 우하단에 **400x600px 점선 박스** 가 상시 존재했다.
완료된 블럭이 여기에 쌓이는 구조였는데, 문제가 분명했다.

- **공간을 너무 많이 차지** — 특히 블럭이 적을 때는 큰 빈 박스가 눈에 거슬림
- **완료 블럭이 쌓이면 그 박스 안을 넘침**
- **캔버스가 잡동사니로 보임**

노션의 좌측 사이드바처럼 **평소엔 숨어있고, 필요할 때만 펼쳐지는** 패턴으로 재설계했다.

```
평소 화면:
┌─────────────────────────┐
│                         │
│        캔버스           │
│                         │
│                   [📦7] │ ← 우하단 박스 아이콘 하나 + 개수 뱃지
└─────────────────────────┘

아이콘 클릭 → 모달 오픈:
┌──────────────────────────────┐
│  갈무리함  (7개 보관됨)        │
├──────────────────────────────┤
│  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐   │
│  │🟩│ │🟦│ │🟨│ │🟪│ │🟧│   │ ← 조약돌처럼 작은 정사각형 타일
│  └──┘ └──┘ └──┘ └──┘ └──┘   │
│  ┌──┐ ┌──┐                  │
│  │🟦│ │🟩│                  │
│  └──┘ └──┘                  │
└──────────────────────────────┘
```

각 타일은 **결 색상 띠 + 시급도 점 + 제목 3줄 클램프**. Hover 하면 "꺼내기 / 삭제" 버튼.

### 드롭 감지를 관대하게

초기 구현에서는 **"블럭의 중심점이 독 bounds ±40px 안"** 을 감지 영역으로 썼다.
그런데 블럭 가로폭이 360px 이라, 블럭을 드래그하면 **중심점이 한참 오른쪽** 에 있다. 사용자가 "박스 위에 뒀는데 갈무리가 안 돼요" 라고 했다.

**블럭 bounds 와 독 bounds 의 겹침(overlap) 검사** 로 바꿨다. **블럭의 어느 귀퉁이라도 독에 닿으면** 갈무리 처리.

```ts
const overlaps = !(
  blockBounds.right < dockBounds.left - padding ||
  blockBounds.left  > dockBounds.right + padding ||
  blockBounds.bottom < dockBounds.top - padding ||
  blockBounds.top    > dockBounds.bottom + padding
)
```

훨씬 자연스러워졌다.

### 꺼내기 시 원래 자리 복원

갈무리 후 다시 꺼냈을 때 **원래 자리로 돌아와야** 한다. 이게 생각보다 까다로웠다.

1. **내가 생각한 첫 번째 이유**: `page.tsx` 의 `handleUpdateBlock` 안에 구 완료 로직이 있어서, `isCompleted: true` 가 들어오면 블럭을 우하단 스택 좌표로 **강제 이동** 시키고 있었다. 이걸 지웠다.
2. **그런데도 안 됐다**. 드래그해서 독에 떨어뜨리면 **마우스 위치(= 독 근처)** 로 x/y 가 저장되고 있었다.
3. **해결**: 드래그 시작 시점의 x/y 를 `dragStartPos` state 에 기억해뒀다가, 독에 드롭되는 순간 x/y 를 **드래그 시작 좌표로 복원** 하면서 `isCompleted: true` 로 전환.

```ts
setDragStartPos({ x: block.x, y: block.y })
// ... 드래그 ...
if (droppedOnDock) {
  onUpdateBlock(draggingId, {
    isCompleted: true,
    x: dragStartPos.x,   // ← 드래그 시작했던 자리로 되돌림
    y: dragStartPos.y,
  })
}
```

이제 꺼내면 **드래그 시작했던 바로 그 자리** 에서 다시 나타난다.

## 6. 오늘의 교훈 4가지

### 1. **"빌드만 되면 되지" 는 최대의 부채**

`ignoreBuildErrors: true` 는 그날의 내가 편하려고 켠 플래그지만, **미래의 내가 갚을 빚** 이었다. 한 번 이 스위치를 켜는 순간 타입 시스템이 내부에서 **조용히 썩어간다**. 그 빚을 오늘 반나절에 갚았다.

### 2. **네이밍이 UX 의 절반**

"완료" 라는 단 두 글자가 **서비스 철학과 싸우고** 있었다. 단어 하나 바꾸니까 — "완료" → "갈무리" — 사용자 질문이 바뀌었다. "이거 다 끝났나?" 에서 "잠시 치워둘까?" 로. 같은 기능, 같은 기술, 다른 단어. **마케팅이 아니라 설계다**.

### 3. **세심한 UX 는 "사용자가 불평하기 전에 고치는 것"**

드롭 감지가 중심점 기반이었던 건 나에게는 **구현하기 쉬운 방식** 이었다. 사용자가 직접 써보기 전엔 몰랐다. **블럭 바운드 겹침** 으로 바꾸는 건 코드로는 5줄이지만 **체감 차이는 크다**.

### 4. **단일 소스의 가치**

시급도 라벨이 두 컴포넌트에서 달랐던 건 **"처음 만들 때 한 곳에 모아두지 않은" 관리 실수**. `lib/constants/urgency.ts` 같은 작은 파일 하나 만드는 게 **나중에 겪을 3배의 혼란을 막아준다**.
