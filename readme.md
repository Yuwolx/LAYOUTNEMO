Visual Canvas-based Work Thinking Tool

LAYOUTNEMO은 기존의 리스트·보드 중심 업무 관리 도구가 담아내지 못했던  
**사고의 흐름과 업무 간 관계를 시각적으로 펼쳐놓기 위한 개인 업무 사고 공간**입니다.

많은 경우, 업무가 정리되지 않는 이유는  
일이 많아서가 아니라 **생각을 담아낼 구조가 없기 때문**입니다.

LAYOUTNEMO은 업무를 “관리”하는 도구가 아니라,  
업무를 **어떻게 인식하고 있는지를 외부에 드러내는 공간**을 만드는 데서 출발했습니다.



## 1. Problem Statement

대부분의 업무 관리 도구는 리스트, 보드, 타임라인과 같은  
**선형 구조를 전제로** 설계되어 있습니다.

하지만 실제 업무 사고는 다음과 같은 특성을 가집니다.

- 하나의 업무가 여러 맥락에 동시에 걸쳐 있음  
- 상·하위 관계가 상황에 따라 계속 변함  
- 되돌아가거나 순환하는 흐름이 잦음  

그럼에도 기존 도구들은 이러한 사고 구조를 담아내지 못하고,  
“정리가 안 되는 문제”를 개인의 관리 능력 문제로 환원시키는 경우가 많습니다.

**이 프로젝트는 이 문제를 기능의 부족이 아니라,  
업무를 담는 구조와 사고 방식의 불일치 문제로 재정의하며 시작되었습니다.**



## 2. Core Philosophy

LAYOUTNEMO는 업무를 ‘정리해야 할 대상’이 아니라,  
**지속적으로 변하는 사고 상태**로 바라봅니다.

사람은 일을 할 때,
- 항상 명확하게 정의된 목표를 가지고 있지 않고
- 모든 업무의 우선순위를 즉시 판단할 수 없으며
- 많은 경우, ‘아직 이름 붙일 수 없는 상태’의 생각을 함께 안고 있습니다.

그러나 대부분의 업무 관리 도구는  
명확한 제목, 분류, 우선순위를 **먼저 요구**합니다.  
이 과정에서 사용자는 생각을 정리하기도 전에  
사고를 압축하거나 생략하게 됩니다.

LAYOUTNEMO는는 이 순서를 뒤집습니다.

먼저 생각을 **그 상태 그대로 펼쳐놓고**,  
정리는 그 다음에 와도 된다고 가정합니다.

그래서 이 서비스는:
- 미완성된 생각을 허용하고
- 애매한 상태를 유지할 수 있으며
- 정리되지 않은 채로 존재하는 것을 실패로 보지 않습니다.

정리는 결과가 아니라 과정이며,  
LAYOUTNEMO에서 ‘정리하기’는 생산성을 높이기 위한 기능이 아니라  
**지금 이 공간이 내 사고와 맞는지 점검하는 체크포인트**로 설계되었습니다.

이 철학은 UX, 데이터 구조, AI 개입 방식 전반에 동일하게 적용되어 있습니다.



## 3. Core Hypothesis

> 업무 관리는 할 일을 나열하는 문제가 아니라,  
> **업무를 인식하고 연결하는 방식을 구조로 표현하는 문제다.**

LAYOUTNEMO의 UX, 데이터 구조, AI 개입 범위는  
모두 이 가설을 기준으로 설계되었습니다.



## 4. Design Principles

### 1. Canvas First
- 정렬보다 **공간적 위치**가 의미를 가짐  
- 사용자는 사고 흐름에 맞게 자유롭게 배치  

### 2. Relationship-oriented Thinking
- 블럭은 독립적인 할 일이 아닌 **사고 단위**  
- 선은 순서가 아니라 **맥락과 연결 상태**를 표현  

### 3. State-based Priority
- 숫자 기반 우선순위 대신  
  **머릿속에서 차지하는 인지적 무게**를 시각적으로 표현  

### 4. User Control & Freedom
- AI는 판단하지 않고 제안만 수행  
- **모든 구조적 변화는 사용자 승인 후 반영**  

### 5. Local-first
- 로그인 없이 즉시 사용 가능  
- 도구 사용 자체의 진입 장벽 최소화  



## 5. Key Features

### Visual Blocks & Connections
- 하나의 블럭은 ‘할 일’이 아니라 **사고 단위**  
- 블럭 간 관계를 선으로 연결하고 언제든 해제 가능  

### Zones (Context Separation)
- 역할·관점·프로젝트 단위로 사고 맥락 분리  
- 영역 삭제 시에도 블럭은 유지 → 사고 기록 보존  

### 5-level State System
- Urgent / Normal / Thinking / Stable / Lingering  
- 우선순위가 아닌 **현재 인식 상태**를 표현  

### Reflection (정리하기)
- 자동 정리 기능 ❌  
- 현재 사고 상태를 점검하는 체크포인트  
- 위치 변경은 항상 사용자 승인 필요  

### Optional AI Assistance
- 자연어 입력 보조  
- 구조적 제안 제공  
- 언제든 ON/OFF 가능  



## 6. AI Design Philosophy

AI는 판단이나 자동화를 담당하지 않습니다.  
LAYOUTNEMO에서 AI의 역할은 다음으로 한정됩니다.

- 입력 마찰 최소화  
- 사고 흐름 단절 방지  
- 구조적 제안 제공  

이를 위해 다음과 같은 구조적 분리를 적용했습니다.

- AI 판단 결과는 `aiMeta`로 분리  
- 사용자 확정 데이터와 절대 혼합하지 않음  
- 위치 변경 등 사고 좌표 영역은 자동 변경 금지  

> AI가 고도화되더라도,  
> 이 도구의 주도권은 항상 사용자에게 남도록 설계되어 있습니다.



## 7. Data Structure as Philosophy

LAYOUTNEMO의 핵심 설계는 데이터 구조 자체에 반영되어 있습니다.

- 보이는 데이터 / 내부 판단 데이터 분리  
- AI 제안 / 사용자 결정 분리  
- 로컬 JSON 직렬화 가능 구조  

이 구조 덕분에:

- 로그인 추가 시 마이그레이션 불필요  
- AI 고도화 시 UI 안정성 유지  
- 기능 확장 시 철학 훼손 방지  



## 8. Role & Responsibility

본 프로젝트에서 수행한 역할은 다음과 같습니다.

- 문제 정의 및 가설 수립  
- 서비스 철학 및 UX 원칙 설계  
- 기능 범위 및 우선순위 결정  
- 데이터 구조 설계  
- AI 개입 범위 및 규칙 정의  
- UI/UX 설계 및 직접 구현  
- 배포 및 운영 환경 구성  

단순한 기능 구현이 아닌,  
**기획–판단–구현을 연결하는 프로젝트 오너 역할**을 수행했습니다.
